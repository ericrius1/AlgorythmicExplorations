The Bounded Dream

Procedural noise is a function with weak imagination. A diffusion model is the opposite: rich imagination trapped inside a rectangle. It can learn statistics from real images, but it usually returns one bounded sample. That is not yet a world.

The paper's central move starts from this contrast. If a diffusion model can learn terrain fidelity and procedural noise can supply open-world utility, the missing mechanism must turn finite learned samples into stable local queries.

Step 1 · Denoising as generationDiffusion in one paragraph

A diffusion model learns to reverse corruption. Training adds noise to examples and teaches a network how to remove a little of that noise. At generation time, the process starts from noise and repeatedly denoises until a sample appears. The model is not following hand-written terrain rules; it has learned the distribution those rules would have to approximate. 

x_T = random_noise(seed)

for t from T down to 0:
  predicted_noise = model(x_t, t, conditioning)
  x_t_minus_1     = denoise(x_t, predicted_noise)

For terrain, the promise is obvious. A model can learn that ridgelines, basins, coastlines, and climate variables have joint structure. It can learn the shape of landscapes rather than only the texture of roughness.

Step 2 · The rectangle problemAn image is not a world

A standard diffusion model has a fixed working area. It denoises a tensor of a particular size, such as a square image or latent grid. That is fine for a poster. It is awkward for a world, because a world engine asks questions like: "What is the next chunk east of the player?" or "What is the biome 50 kilometers away?"

The naive answer is to generate independent tiles. That fails because the border of one tile did not know the border of the next. Even if both tiles look plausible alone, their ridges, rivers, and climate bands will disagree at the seam. 

tile_A = model(seed_A)
tile_B = model(seed_B)

// Both are plausible.
// Neither promised continuity with the other.

Step 3 · Overlap helpsMultiDiffusion

MultiDiffusion is the bridge idea. Instead of asking one model for one whole image, slide the model over many overlapping windows. Each window predicts a denoising update for its local patch. Where windows overlap, their predictions are blended. 

for each window W_i:
  local_update = denoise_window(x inside W_i)
  add local_update into the global canvas
  add blending_weight into the global denominator

x_next = numerator / denominator

This is powerful because local windows can be fused into a larger coherent image. A feature that crosses a window boundary is seen by more than one denoising path, so the boundary becomes a blending zone rather than a cut.

But ordinary MultiDiffusion still assumes there is a finite canvas whose numerator and denominator can be stored. It stretches the rectangle. It does not yet remove the rectangle.

Step 4 · What the world engine needsQueries, not posters

A game or simulation does not need a whole infinite image. It needs a region around the player, then a neighboring region, then perhaps a faraway region after teleportation. The correct abstraction is not "generate everything." It is "answer this bounded query consistently."

That distinction is the key to InfiniteDiffusion. If a fixed-size query only depends on a finite set of overlapping windows, then the fact that the mathematical world is unbounded does not mean the computation is unbounded. It means the generator has to find and evaluate only the windows that matter.

Aside · The edge moves from pixels to logic

Independent tiles have visible pixel edges. MultiDiffusion hides many of those edges by overlapping windows. InfiniteDiffusion goes further: the output region can be anywhere, so the remaining boundary is the query itself. If the next query overlaps or touches it, both queries must agree because they are slices of the same implicit sample.

CodaStretching is not enough

Diffusion gives terrain a learned prior. MultiDiffusion shows how many local denoising paths can be fused. The missing step is procedural behavior: seed-stable local access on an unbounded domain. The next post is that step.
