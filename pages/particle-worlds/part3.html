Sorting Water

Part three: the particles stop attracting and start colliding. Counting sort, parallel prefix sums, and a glass of water in your browser tab. Stir it with your cursor.

Scroll down to read ↓

Part one built a galaxy: hundreds of thousands of bodies, every one attracting every other, made tractable by a tree that summarized faraway crowds into single points. Part two lifted it into 3D and bent it over a dome. Both addressed the same constraint: gravity reaches across the entire universe, so in principle everything matters to everything.

The liquid sloshing above is a different world. Water doesn't pull on water across the tank. A drop only cares about the drops it is actually touching — push on your neighbours, be pushed back, and the sum of tens of thousands of tiny shoves becomes pressure, splashes, and waves. Collisions, flocking birds, and molecular dynamics all work this way. Forces with a cutoff radius are everywhere in simulation, and they sound easier: surely "check a few neighbours" is easier than "sum the whole universe"?

It is — once you know who your neighbours are. Answering that question at GPU speed takes one of the oldest algorithms in computing plus a fundamental parallel primitive. Together they make the fluid demo possible.

Step 1 · The opposite problemWasted pair checks

Start with a box of discs under gravity. Two discs push apart when they overlap — a stiff spring plus a little damping, the standard contact model — and feel nothing otherwise. The obvious implementation is the same two nested loops as the galaxy: 

for (let i = 0; i < count; i++) {
  for (let j = i + 1; j < count; j++) {
    if (distance(i, j) < 2 * radius) {   // almost always false
      applyContactForce(i, j);
    }
  }
}

Run it and watch the middle number in the counter below. At 2,000 particles the loop performs about two million distance checks per step, and on a typical frame fewer than three thousand find discs actually touching. With gravity, every term of the O(n²) sum contributed something. Here, more than 99.8% of the work computes a force of exactly zero.

Poke the pile — the cursor shoves nearby discs, and the shove costs the same wall of distance checks. The fix can't be the part-one fix: a Barnes–Hut tree answers "what does everything far away add up to?", and here the far field adds up to nothing by definition. We need the mirror-image structure — one that answers "who is near me?" and skips the rest entirely.

Step 2 · BucketsThe spatial hash grid

The classic answer is direct. Lay a uniform grid over the world, with cells exactly one interaction-radius wide, and drop each particle into its cell. Any particle close enough to touch me must sit in my cell or one of the eight surrounding it — by construction, since a cell is as wide as my reach. The neighbour query becomes: look up nine buckets, test their occupants, done. For roughly uniform particles that is O(n) total.

Aside · The same idea across fields

Molecular-dynamics people have leaned on this since the field began — cell lists and Loup Verlet's neighbour lists date to the 1960s, when simulating a few hundred argon atoms was worth a paper in Physical Review. Game engines call it spatial hashing, graphics calls it a uniform grid, databases call it binning. Every field that needed nearness rediscovered it.

The demo below runs the same pile of discs, now grid-accelerated. Your cursor performs the query live: the yellow 3×3 window is every cell it searches, the yellow discs the only candidates it tests. Move toward a dense clump and the candidate count rises to a few dozen; the other 3,500 discs might as well not exist.

A CPU implementation is almost suspiciously small. Convert each position to one integer cell key, append particle indices to that bucket, then visit the 3×3 block around the particle: 

const buckets = Array.from({ length: grid * grid }, () => []);

for (let i = 0; i < particles.length; i++) {
  buckets[cellOf(particles[i].position)].push(i);
}

function testNeighbours(i) {
  const [cellX, cellY] = cellCoord(particles[i].position);
  for (let offsetY = -1; offsetY <= 1; offsetY++) {
    for (let offsetX = -1; offsetX <= 1; offsetX++) {
      const cell = cellAt(cellX + offsetX, cellY + offsetY);
      if (cell === undefined) continue;           // outside the box
      for (const j of buckets[cell]) {
        if (j > i) testPair(i, j);                // each pair once
      }
    }
  }
}

The query is solved; the innocent-looking push is the GPU obstacle. Appending to a list means pointers, allocation, and a write position that depends on who appended before you — the same serial dependency that complicated the Barnes–Hut tree build. Ten thousand threads cannot politely take turns appending. Whatever structure we build must be built the GPU way: every thread at once, no one waiting.

Part one escaped this trap by making the tree implicit — pure arithmetic, nothing to link. For the grid, sorting provides the equivalent layout.

Step 3 · Ordering the gridCounting sort and the parallel for-loop

Notice what we actually need. Not lists — just each cell's particles sitting together, plus, for every cell, the index where its group starts. That is precisely what a sort delivers: order the particles by cell number, and each cell becomes a contiguous run in one flat array. The pointer structure we couldn't build turns out to be an ordering, and orderings are something GPUs can produce.

Better still, sorting by bucket number needs no comparisons. The recipe is counting sort: count how many particles land in each cell; compute, for every cell, the total count of all cells before it — that running total is exactly where the cell's group starts; then move every particle directly to its slot. 

counts[cell]++                       // 1. histogram      (one pass)
starts[c] = counts[0..c-1] summed    // 2. prefix sum     (the interesting bit)
slot = starts[cell] + claimed++      // 3. scatter        (one pass)

In serial JavaScript, the whole sort is the following. The GPU version keeps the same arrays and the same three phases; only the loops become compute dispatches and the two increments become atomics: 

const counts = new Uint32Array(cellCount);
for (const p of particles) counts[cellOf(p)]++;          // histogram

const starts = new Uint32Array(cellCount);
for (let c = 1; c < cellCount; c++) {
  starts[c] = starts[c - 1] + counts[c - 1];             // exclusive scan
}

const cursor = starts.slice();
const sorted = new Array(particles.length);
for (const p of particles) {
  const cell = cellOf(p);
  sorted[cursor[cell]++] = p;                             // scatter
}

Step 2 — turning per-cell counts into running totals — looks like the one inherently serial part. A running total seems to demand a single accumulator marching left to right: you can't know the total through cell 100 until you've finished cell 99. How could a thousand threads possibly help?

The answer, from Hillis and Steele in 1986, works by doubling. The scan runs in rounds, and in each round every cell updates at once. Round one: each cell adds in the value of its immediate left neighbour, so each cell now holds the sum of two cells — itself and the one before. Round two: each cell reaches two positions left and adds that value in. But that value already covers two cells from round one, so every cell now holds the sum of four. Round three reaches back four positions, and each cell covers eight. The reach doubles each round, and so does the stretch of array each cell has absorbed — after log₂ n rounds (ten rounds for a thousand cells), every position holds the sum of everything up to it. Within a round no thread waits on any other; the serial loop becomes a handful of perfectly parallel sweeps.

Aside · Census machines and a refrigerator-sized computer

Counting sort predates electronic computers by half a century. The 1880 US census took eight years to tabulate by hand; facing worse in 1890, the Census Bureau bought Herman Hollerith's punched-card tabulating machines, which grouped and counted cards by category, bucket by bucket. Clerks sorting cards digit-by-digit into bins were executing radix sort decades before anyone named it — and Hollerith's company, after some mergers, became IBM. Harold Seward's 1954 MIT thesis finally wrote down the algorithm the punched-card world had been running for sixty years.

The parallel scan has its own pedigree: Hillis and Steele were programming the Connection Machine, a computer built from 65,536 one-bit processors — and our grid happens to be 256×256 = 65,536 cells, one per processor.

This operation — scan, or prefix sum — turned out to be far bigger than sorting. Guy Blelloch's work at Carnegie Mellon recast scan as a universal primitive: radix sort is scans, stream compaction is a scan, memory allocation across threads is a scan. The slogan that stuck: scan is the parallel for-loop — whenever a loop carries a running quantity forward, a scan may be able to run it in logarithmic depth instead.

Deep dive: the scan as three GPU dispatches

On a modern GPU, scanning 65,536 cells takes three dispatches: each 256-thread workgroup scans its own 256 cells in shared memory; one workgroup scans the 256 block totals; a final sweep adds each block's total back onto its cells. About a tenth of a millisecond, all told. 

pass.setPipeline(scanBlocks);  pass.dispatchWorkgroups(256);
pass.setPipeline(scanSums);    pass.dispatchWorkgroups(1);
pass.setPipeline(scanAdd);     pass.dispatchWorkgroups(256);

Inside the first two dispatches, each 256-thread group performs a Hillis–Steele scan in fast workgroup memory. The offset doubles after every barrier, so all threads read the previous round before any thread starts the next: 

var<workgroup> values: array<u32, 256>;
var<workgroup> scratch: array<u32, 256>;

fn scan(lid: u32) {
  var offset = 1u;
  loop {
    if (offset >= 256u) { break; }

    var next = values[lid];
    if (lid >= offset) { next += values[lid - offset]; }
    scratch[lid] = next;
    workgroupBarrier();

    values[lid] = scratch[lid];
    workgroupBarrier();
    offset = offset << 1u;
  }
}

Step 4 · AssemblyThe GPU implementation

The complete GPU pipeline runs every simulation step from scratch because the particles never stop moving: 

clear    → zero the cell counters
count    → every particle atomically bumps its cell    (histogram)
scan     → starts[] from counts[]                      (3 dispatches)
scatter  → every particle copies itself to its slot    (one atomicAdd each)
forces   → every particle reads its 9 cells, pushes, integrates

The histogram and scatter kernels are each only a few lines of WGSL. Atomics do not impose an order; they only guarantee that no two particles receive the same ticket: 

@compute @workgroup_size(256)
fn count(@builtin(global_invocation_id) gid: vec3u) {
  if (gid.x >= GP.count) { return; }
  atomicAdd(&counts[cellOf(partsIn[gid.x].xy)], 1u);
}

@compute @workgroup_size(256)
fn scatter(@builtin(global_invocation_id) gid: vec3u) {
  if (gid.x >= GP.count) { return; }
  let p = partsIn[gid.x];
  let slot = atomicAdd(&cursor[cellOf(p.xy)], 1u);
  sorted[slot] = p;
}

Two details matter here. The scatter copies each particle's entire state into the sorted array, not just an index — so after the sort, particles that are neighbours in space are neighbours in memory, and the force pass reads clean contiguous runs instead of pointer-chasing. (Part one's epilogue promised memory coherence as a future optimization; here the sort provides it directly.) The only coordination is atomic increments, which the hardware resolves without anyone stopping.

Deep dive: losing the box — spatial hashing

One assumption matters here: our grid is dense — 65,536 cells covering a fixed box, most of them empty, each with its own counter. For a tank with walls that's the right call. But nothing about the method needs walls. Replace "cell index" with a hash of the cell coordinates — multiply ix and iy by two large primes, XOR, mask to the table size — and the same count-scan-scatter pipeline runs over an unbounded world: only cells that actually contain something cost anything.

Distant cells occasionally hash to the same slot, and the algorithm doesn't care: colliding cells share a bucket, and the distance test in the force pass — always the final arbiter of who counts as a neighbour — silently discards the impostors. This formulation (Teschner et al., 2003) is what you reach for the moment your particles outgrow their box, and part four will need it immediately.

Step 5 · From sand to waterSmoothed particles

Sand grains only interact when they collide. Water needs more: it resists compression everywhere, all the time, even in the calm middle of the tank. The technique is smoothed particle hydrodynamics: represent the fluid as particles, but treat each one as a small blurry blob of matter rather than a point. The density anywhere is the sum of the blobs overlapping that spot. Density above the resting value means squeezed, and squeezed fluid pushes its neighbours apart; below means stretched, and it pulls back. Pressure becomes one more short-range force, so the neighbor-search pipeline handles it unchanged. 

// pass 1 — density: how crowded is each particle?
ρ_i = Σ over neighbours j:  (1 − r/h)²

// pass 2 — pressure force: push from crowded toward roomy
P_i  = k · (ρ_i − ρ_rest)
acc -= r̂ · (P_i + P_j)/2 · (1 − r/h)

The sort turns “over neighbours” into two ordinary integer ranges. This is the inner loop of the density pass: visit nine cells, read each contiguous run, then reject candidates outside the smoothing radius. 

for (var offsetY = -1; offsetY <= 1; offsetY++) {
  for (var offsetX = -1; offsetX <= 1; offsetX++) {
    let cell = cellCoord(pos) + vec2i(offsetX, offsetY);
    if (cell.x < 0 || cell.y < 0 ||
        cell.x >= i32(SP.grid) || cell.y >= i32(SP.grid)) { continue; }

    let key = u32(cell.y) * SP.grid + u32(cell.x);
    let first = cellStart[key];
    let end = first + cellCount[key];

    for (var j = first; j < end; j++) {
      let delta = parts[j].xy - pos;
      let distSq = dot(delta, delta);
      if (distSq < h2) {
        let weight = 1.0 - sqrt(distSq) / h;
        rho += weight * weight;
      }
    }
  }
}
Aside · Invented for stars, ended up in Moana

SPH was invented by astrophysicists. In 1977, Leon Lucy — and independently Gingold and Monaghan — needed to simulate gas collapsing into stars, flows so distorted that any fixed mesh would tangle. After twenty years in astrophysics the method crossed into engineering (coastal flooding, fuel sloshing) and then into every film and game studio that needed splashes. Interactive SPH on consumer hardware traces to Matthias Müller's 2003 paper, a direct ancestor of what's running below. Big-budget film water has since moved to hybrid grid-particle methods like FLIP — which need the same ingredient this article built: particles, binned to a grid, every frame, fast.

Deep dive: the stabilizers that keep game SPH from clumping

Two passes instead of one — density first, then forces — plus two stabilizers from the games literature. A short-range "near pressure" that is always repulsive (Clavet, Beaudoin and Poulin's 2005 viscoelastic-fluids paper) keeps particles from clumping into strands. A touch of velocity smoothing between neighbours reads as viscosity.

Schemes where everything reduces to forces along the line between particle pairs form their own branch of the SPH literature — pairwise-force SPH — prized because pair forces conserve momentum exactly, by Newton's third law, no matter how crude the rest of the model is. The walls of the tank are three penalty springs: part two's dome trick, constraints expressed as forces.

That is a dam break — a classic test problem of computational fluid dynamics — at tens of thousands of particles, each one sorted, densified, pressurized and integrated five times per frame. Stir it. Make waves. Crank the particle count and watch the water get finer rather than slower, for a while — resolution, as in part one, is just a slider.

EpilogueResults and tradeoffs

1. On a GPU, data structures become orderings. The CPU's per-cell linked lists were unbuildable in parallel; sorting the particles by cell produced the same information as pure layout. Part one's pyramid replaced pointers with arithmetic; counting sort replaces them with a permutation. When a structure won't parallelize, ask what ordering would make it unnecessary.

2. Scan is the parallel for-loop. One log-depth primitive unlocked sorting here, and it unlocks compaction, allocation, and a hundred other "but that part is sequential" complaints elsewhere. It is the single highest-leverage algorithm in GPU computing.

3. Physics is mostly data movement. Count the dispatches in the final pipeline: five move and organize data, two compute forces. The actual physics — a spring, a pressure gradient — is a few lines; the achievement is arranging memory so those lines read at full bandwidth. This ratio is normal, and learning to see it is most of GPU programming.

4. The far field and the near field want opposite structures. Gravity needed a hierarchy, because distant things still mattered in aggregate. Contact forces needed flat buckets, because distant things didn't matter at all. Production codes often need both at once — and now you've built each from scratch.

The same approach extends in several directions. Three dimensions is the same sort with a third coordinate folded into the cell index. Radix sort — counting sort applied digit by digit — handles keys when cells alone aren't enough. And nothing stops both structures from living in the same simulation: part one's mass pyramid for the far field, this article's hash grid for the near field, gravity from one and contacts from the other. That pairing is the beginning of a planet-formation simulator — dust attracting, colliding, and sticking.
