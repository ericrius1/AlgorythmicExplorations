Fireworks at GPU Scale

A fireworks renderer is a lesson in where not to put the CPU. The useful architecture is not a bigger queue or a clever emergency reserve; it is a frame graph where the CPU describes intent and the GPU owns the millions of small glowing facts.

Scroll down to read ↓

Fireworks are simple in the way that performance problems are simple: every individual piece is cheap, and there are suddenly too many pieces. A rocket is one particle. Its smoke trail is a few more. The burst is hundreds, then thousands, then tens of thousands of sparks. Click once and nothing strains. Hold the mouse down, raise the spark count, ask for many rockets in the same half second, and the shape of the program matters more than the constants.

The obvious first design fails in the ordinary way. Keep a large ring buffer of particles. Launching a firework means the CPU writes particle records into that ring; old particles expire by time; the renderer draws the portion of the ring that has been touched. That sounds reasonable until the test case becomes deliberately unreasonable: many fireworks close together, very high spark counts, and no interest in protecting a small laptop from itself. Then older sparks disappear because newer sparks lap the ring. Or the opposite happens: nothing visible remains, but the debug overlay still reports hundreds of thousands of active particles, because the CPU's bookkeeping and the GPU's visual truth have drifted apart.

The fix is not to add a hidden reserve, a second emergency buffer, or a more elaborate policy about which old sparks deserve to live. The fix is to remove that policy. A visual effect with this many independent particles should not ask the CPU to manage individual particles in the first place.

This article is the architecture that follows from that constraint. It borrows the frame-graph mindset from Worlds from Dust: fixed GPU buffers, explicit passes, tiny CPU parameter writes, and a renderer that reads the buffer the simulation just produced. The physics here is not planetary science. It is art-directed sparks, smoke, gravity, drag, curl, and color. But the performance philosophy is the same.

Step 1 · The wrong unit of workEvents, not particles

A user does not create 50,000 particles. A user launches a firework. The firework happens to contain 50,000 sparks, but that count is an implementation detail of the effect, not the semantic event. If the CPU uploads one record per spark, the CPU is doing work at the wrong level of abstraction.

The better unit is an emit command. A command is a compact description of a family of particles: 

struct EmitCommand {
  header:       vec4u,  // kind, count, start, aux
  posScaleSeed: vec4f,  // origin, size scale, random seed
  velLifeSize:  vec4f,  // velocity or burst speed, lifetime, size
  color:        vec4f,
  accent:       vec4f,
  homeParams:   vec4f,  // target/home, depth, extra counts
}

A rocket command creates one particle. A trail command creates a handful of smoke and ember particles behind the rocket. An explosion command creates the center flash, the main shell, crackle particles, falling embers, and smoke. The command does not contain those particles; it contains the recipe. The GPU runs one compute invocation per emitted particle, derives a local index inside the command, hashes the command seed with that local index, and procedurally fills out the particle record.

That changes the CPU's job from "write every spark" to "write one command per meaningful event." The difference is not subtle:

The numbers in the diagram are intentionally boring. A particle record in the current renderer is 80 bytes: position, velocity, color, age/life/size/kind, seed/level/depth, and a home anchor. An emit command is 96 bytes. For one firework with 50,000 sparks, uploading particles is roughly 3.8 MiB of CPU to GPU traffic. Uploading the command is 96 bytes. For a burst of many fireworks, the ratio becomes absurd enough that it stops feeling like optimization and starts feeling like type correctness.

The GPU still does the particle work. It still creates every spark. It still writes every particle record. But that work happens in parallel, adjacent to the buffer that will be simulated and rendered. The CPU stays at the scale of interaction: pointer down, launch count, target position, colors, spark count.

Aside · Randomness belongs to the command

A command needs a seed, not a list of random numbers. The local particle index provides the second coordinate. The shader can derive angle, speed, lifetime, shell level, smoke drift, and color interpolation from hash(seed, local). This is why the CPU can describe an explosion without enumerating the explosion.

Step 2 · The GPU owns the stateA ping-pong particle world

Once emission happens on the GPU, the rest of the architecture becomes easier to see. There is no CPU particle array. There is no CPU list of expiration times. There is no high-water mark that guesses how much of a ring buffer is worth drawing. There are two GPU particle buffers: 

particleBufferA: array<Particle>
particleBufferB: array<Particle>

On frame N, one buffer is the read buffer and the other is the write buffer. The simulation reads every live particle from the old buffer, advances it, and appends survivors to the new buffer. New particles from emit commands append to the same new buffer. At the end of the frame, the new buffer becomes the current state. Next frame the roles swap.

This is the same ping-pong pattern used by fluid solvers, cellular automata, particle-mesh codes, and the accretion demos earlier in the particle series. It has one beautiful property: the write buffer is always compact. If a spark dies, it simply is not appended. No hole remains. No old slot needs to be zeroed. No CPU cleanup pass waits around to discover that the scene is empty.

The compaction pass is short enough to write as a sketch: 

@compute @workgroup_size(256)
fn simulateCompact(@builtin(global_invocation_id) gid: vec3u) {
  let i = gid.x;
  if (i >= counters.live) { return; }

  var p = simulate(particlesIn[i]);
  if (p.age >= p.life) { return; }

  let out = atomicAdd(&counters.write, 1u);
  particlesOut[out] = p;
}

The important line is the atomic add. It turns a sparse set of survivors into a dense write list without asking any invocation where the others went. Every live particle claims the next slot. Dead particles claim nothing. If a burst creates 200,000 particles and 148,000 survive the frame, the write counter says 148,000. That number is not inferred later; it is the result of building the list.

This also resolves stale active counts. The active count is no longer a CPU-side estimate derived from timestamps. It is the GPU's own write counter, copied to a tiny readback buffer for the debug overlay. When the last spark dies, the next compaction pass appends zero particles and the active count becomes zero. The renderer and the readout are looking at the same fact.

Step 3 · No draw loop bookkeepingIndirect rendering

If the CPU does not know how many particles survived, how does it draw them? The GPU writes the draw call arguments too.

A particle is rendered as a six-vertex billboard: two triangles stretched in the direction of velocity, shaded as a soft additive blob. WebGPU's indirect draw buffer for this renderer is four integers: 

drawArgs.vertexCount   = 6
drawArgs.instanceCount = liveParticleCount
drawArgs.firstVertex   = 0
drawArgs.firstInstance = 0

The finish pass writes that structure after simulation and emission have appended their particles. The render pass then calls drawIndirect(drawArgs). The CPU does not read back the count, wait for it, and call draw(6, count). It records the same command every frame and lets the GPU fill in the count before the command executes.

The whole frame is a small command buffer: 

beginFrame       // reset write counter, overflow counter, draw args
simulateCompact  // read old buffer, append live survivors
emitParticles    // expand compact commands into new particles
finishFrame      // publish live count, draw args, next dispatch size
renderScene
drawIndirect     // render exactly the compacted live list

The pass order matters. New particles are appended after old survivors, so a particle emitted this frame renders this frame. The next frame's indirect dispatch size is also written on the GPU, so the simulation does not need to dispatch across the full capacity every time. It dispatches over the live count from the previous finish pass.

That last sentence is one of the quiet performance wins. The buffer may have capacity for more than a million particles, but an idle frame does not scan a million empty slots. It dispatches zero workgroups for simulation, maybe a few for emission, writes a draw count of zero, and moves on.

Step 4 · Capacity is not a vibeMemory as a first-class readout

Removing artificial limits does not mean pretending hardware has no limits. It means distinguishing a policy limit from a device fact. "Never let more than 150,000 particles exist because performance might be bad" is a policy. "This adapter allows a 128 MiB storage buffer binding" is a fact. A debugging tool should show facts.

The current particle stride is 80 bytes. The maximum particle capacity is therefore: 

capacity = floor(device.limits.maxStorageBufferBindingSize / 80)

On a typical desktop GPU, the storage buffer binding limit reported by the browser is 128 MiB, which gives 1,677,721 particles per particle buffer. The fireworks system keeps two such buffers, because compaction ping-pongs from old state into new state. That is why the debug overlay's "storage limit" and "fireworks GPU memory" do not match. The limit is per binding. The total allocation includes both particle buffers, the emit command buffer, counters, indirect argument buffers, and uniforms.

This is the honest version of "no artificial limiting." The capacity is as large as the current WebGPU device allows for one particle buffer. The overlay reports the capacity, total fireworks GPU allocation, overflow count, live particles, and active launches. If the user chooses settings that exceed the buffer, the shader increments an overflow counter. It does not silently erase old particles to make room for new ones. It does not maintain a secret emergency reserve. It tells you what happened.

Aside · Total app memory is not exposed as a promise

Browsers expose WebGPU device limits such as maximum storage buffer binding size. They do not generally give a portable, reliable "total memory this app may use" number. For this renderer, the useful diagnostic is therefore the storage binding limit plus the actual GPU allocations made by the fireworks system. The first tells you the largest legal particle buffer; the second tells you what the effect currently reserved.

Step 5 · Simulation is cheap when it is localThe actual particle physics

None of the above says the particles are visually simple. Each particle still carries a kind: rocket, spark, ember, or smoke. Each kind gets different acceleration, drag, size, color, parallax depth, and lifetime. Rockets ignore most gravity so they can hit their target. Sparks fall and curl. Smoke rises and fattens. Embers live longer and drift down through the scene.

The shader also adds an art-directed curl field. A small grid-like noise function turns position and time into a direction; that direction becomes wind, turbulence, and shell swirl. It is not a Navier-Stokes solve. It is the cheaper lesson from the fluid instrument: if the eye needs curling motion, a divergence-free-ish procedural field often gives more visual value per millisecond than a full fluid solve.

The renderer then makes physics legible. Each particle becomes a billboard stretched along velocity. Fast sparks become streaks; slow smoke becomes soft round puffs. Depth is sampled from the painted mountain scene so sparks can pass behind terrain silhouettes. Additive blending does what fireworks want: many dim particles accumulating into a bright burst.

The important performance property is locality. A particle update does not need to find neighbors, sort a grid, solve a pressure field, or walk a tree. It reads its own record, reads uniforms, writes one survivor or nothing. High counts are possible because the expensive part is embarrassingly parallel.

Step 6 · The boring parts are the sharp partsWGSL layout discipline

The most useful discipline here is not glamorous: make storage layouts explicit. It is tempting to address counters and indirect arguments as bare indexed storage arrays: 

counters[0]  // active count?
counters[1]  // write count?
counters[4]  // overflow?

That is compact, but it hides meaning at exactly the layer where meaning matters. Named structs make the shader state the contract: 

struct Counters {
  live:         atomic<u32>,
  write:        atomic<u32>,
  commandCount: atomic<u32>,
  totalEmit:    atomic<u32>,
  overflow:     atomic<u32>,
}

The same is true for draw and dispatch indirect arguments. A buffer is still only bytes, and the CPU still writes words at fixed offsets, but the shader names the contract. That makes validation errors easier to reason about and removes a class of "which counter was index 3?" mistakes.

Explicit layouts also surface a very WebGPU kind of problem early: intuitive field names like active and target are reserved WGSL words in some browser validators. A shader that uses them does not fail at the line where the render loop binds the pipeline; it fails earlier, during shader module validation, after which every frame reports "invalid pipeline." The practical rule is to ask for the root error before the frame loop floods the console: 

device.pushErrorScope("validation");
const module = device.createShaderModule({ code });
const error = await device.popErrorScope();
const info = await module.getCompilationInfo();

That is not a fireworks trick. It is a WebGPU habit. Build shader modules and pipelines with labels. Use error scopes around suspicious creation calls. Read compilation info. A fast simulation that cannot explain its own invalid state is not a fast simulation; it is just opaque.

Step 7 · Diagnostics without synchronizationRead back less than you want

The debug overlay is intentionally small. It shows FPS, frame time, active particles, particle capacity, draw slots, active fireworks, queued bursts, particles emitted this frame, overflow, fireworks GPU memory, and the storage binding limit. That is enough to answer the questions that came up during testing:

Are particles disappearing because the capacity was exceeded? Check overflow. Is the app slow because hundreds of thousands of invisible particles remain active? Check active particles after the effect has faded. Is draw work still happening when the scene is empty? Check slots drawn. Is the cap a magic constant or a hardware limit? Compare capacity to storage limit.

The overlay does not read back the particle buffer. It does not inspect individual sparks. It does not synchronize every frame. Every few tenths of a second, it copies the small counters buffer into a map-readable buffer and updates the UI when the mapping completes. The simulation never waits for the overlay. Diagnostics are observers, not part of the hot path.

Aside · Metrics change behavior if you let them

A common trap in GPU demos is to add a helpful readback that accidentally serializes the pipeline. Reading a million particles to count them is worse than not knowing. Maintaining a counter on the GPU and occasionally reading eight integers is the difference between instrumentation and interference.

Step 8 · Why the solar-system demo matteredOne frame as a graph

The strongest influence from the particle series was not a specific force law. It was the shape of the accretion demo in part four. That demo combines gravity and collisions by building GPU data structures in phases: sort into a spatial hash, reduce into a gravity pyramid, run a force pass, render the resulting buffer. The CPU writes parameters and records passes; the GPU owns the evolving particle state.

Fireworks are physically simpler, so they do not need a tree or a neighbor grid. But they benefit from the same mental model: 

resources are long-lived
passes have explicit read/write ownership
the CPU writes parameters and commands
the GPU transforms one valid state into the next valid state
the renderer consumes the state that the simulation just produced

This is why "just use a much bigger ring buffer" is only half right. A bigger buffer postpones overwrite bugs. It does not answer who owns liveness, how dead particles disappear, where the draw count comes from, or whether the CPU should be writing individual sparks. The real simplification is replacing the ring with a current-state buffer and a next-state buffer. Once the state is a compact list, old particles cannot be overwritten by new ones. They either survive into the next list or they do not.

Step 9 · PhilosophyRemove policy, keep physics

The guiding principle was not "optimize everything." It was more specific: remove artificial policy from the hot path and make the remaining constraints visible.

A maximum particle count chosen because it feels safe is policy. A hidden reserve for emergencies is policy. Dropping old particles because new ones arrived is policy. Drawing a stale high-water range because compaction is inconvenient is policy. The GPU storage binding limit is not policy. Particle stride is not policy. A user's spark slider is not policy. Those are facts or explicit choices.

The final architecture tries to keep the facts close to the code: 

capacity       = floor(storageLimit / particleStride)
liveCount      = GPU write counter
overflow       = GPU overflow counter
drawnInstances = indirect draw instance count
memory         = actual buffers allocated by this system

There is still a limit, because buffers are finite. But it is now the real limit. There is still an overflow case, because a user can request more particles than fit. But overflow is counted, not disguised as "older sparks gracefully disappeared." There is still performance cost, because millions of billboards are millions of billboards. But the cost is where it belongs: in GPU work proportional to live visible particles, not in CPU bookkeeping proportional to every particle ever requested.

EpilogueWhat I would add next

This architecture is a good base, not an ending. Several extensions fall out naturally.

GPU-side command generation. Some secondary effects are still scheduled by the CPU at the firework level: satellite bursts, follow-up pops, and timing choices. A more autonomous system could let compute passes append future emit commands into a command buffer, turning the whole firework script into GPU state. That would need a second command queue and careful lifetime rules, but the pattern is the same as particle append.

Depth-aware sorting or layered blending. Additive sparks do not require exact sorting, but smoke and darker particles benefit from better order. A coarse depth binning pass could sort only the particles that need it, while hot additive sparks remain unordered.

Timestamp profiling. The overlay currently reports frame time, not pass time. WebGPU timestamp queries would separate simulation, emission, scene rendering, and particle rendering. That matters when a high spark count drops FPS: the next question is whether the limit is compute emission, compaction, fragment fill, or blending.

More authored commands. The command idea scales beyond fireworks: fountains, sparks from collisions, rain splashes, magic spells, dust kicked off terrain, debris from explosions. If the event can be described compactly and expanded procedurally, the CPU should probably send the event, not the particles.

The main lesson is smaller than the code. High-performance browser graphics usually comes from asking which facts need to cross the CPU/GPU boundary at all. For fireworks, the answer is: far fewer than the obvious first design assumes. A click, a target, a seed, a count, a color. The rest is light.
