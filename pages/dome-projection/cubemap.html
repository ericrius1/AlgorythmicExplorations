Cubemap or Fisheye Master

Part two described a fisheye master as a lookup table for directions. A cubemap is another lookup table for the same thing. The format changes; the underlying object does not. The stable object is still a normalized direction vector from the dome center.

This choice matters because it changes the render path, the preview path, the calibration path, and sometimes the performance budget. A fisheye is usually the delivery format for fulldome video. A cubemap is often the easier render target for a realtime 3D engine. A working dome system may use both.

Same directionTwo storage layouts

Start with a direction, not an image format. If a shader asks for the color at one point on the dome, it first finds the direction from the dome center to that point. After that, it can ask a fisheye master or a cubemap for a color. 

dir = normalize(domePoint - domeCenter)

fisheye lookup:
  radius  = projectionLaw(zenithAngle(dir))
  angle   = atan2(dir.y, dir.x)
  uv      = radiusAngleToUv(radius, angle)

cubemap lookup:
  face    = majorAxis(dir)
  uv      = directionToFaceUv(dir, face)

In the demo, drag either blue handle. The point moves in both formats because both are just different addresses for the same direction.

Fisheye masterChoose it when delivery matters

A fisheye master stores the dome as one circular image. The center is the zenith. The circular edge is the horizon for a 180 degree dome. The angle around the image is azimuth. 

good fit:
  fulldome video files
  one-projector fisheye lens setups
  venue systems that expect a dome master
  easy monitor preview and recording
  offline rendering that will tour between domes

The advantage is practical compatibility. A fisheye file is a compact artifact you can hand to another dome pipeline. Calibration software can warp that one master into whatever projector layout the room uses.

The tradeoff is that a fisheye is a specialized projection. Sampling is dense near some regions and stretched in others depending on the fisheye law. The center is a polar singularity. The circular edge and seam need deliberate testing. For realtime 3D, rendering a true fisheye directly may require a custom shader or a conversion pass.

CubemapChoose it when rendering the 3D scene is the hard part

A cubemap stores directions as six square faces. Each face is a normal perspective camera with a 90 degree field of view. For an upper dome you usually care about the top face and the upper halves of the four side faces. A full sphere uses all six. 

good fit:
  realtime Three.js scenes
  environment maps and reflections
  engines built around perspective cameras
  direct sampling in a projector warp shader
  conversion to several output formats from one render target

The advantage is that renderers already know how to draw perspective cameras. Shadows, culling, post effects, and material systems tend to be easier to reason about in 90 degree chunks than in one 180 degree fisheye view.

The cost is render count and seams. A cubemap is often five or six scene renders before you start the projector passes. Faces must match at their borders. Screen-space effects such as bloom, motion blur, temporal antialiasing, and SSAO can reveal face edges if they are applied per face without care.

ConversionWhere the extra pass appears

Cubemap and fisheye are not opposing religions. They are often two stages in one pipeline. A realtime scene can render to a cubemap, then a shader can sample that cubemap into a fisheye master for recording or venue playback. The same cubemap can also be sampled directly by each projector calibration mesh. 

cubemap -> fisheye:
  for each fisheye pixel:
    dir   = fisheyeUvToDirection(uv)
    color = sampleCubemap(dir)

fisheye -> projector:
  for each projector pixel:
    dir   = calibratedPixelToDomeDirection(pixel)
    uv    = directionToFisheyeUv(dir)
    color = sampleFisheye(uv)

cubemap -> projector:
  for each projector pixel:
    dir   = calibratedPixelToDomeDirection(pixel)
    color = sampleCubemap(dir)

The last path skips the circular master, but it does not skip texture sampling. It changes which texture is sampled.

ExamplesWhen to choose each one

Festival submission or touring fulldome film

Choose fisheye. The file needs to travel. A venue can map a fisheye master through its own calibration system without knowing your scene graph or projector layout.

Realtime Three.js world with meshes, shadows, and particles

Choose cubemap first. Six 90 degree cameras match the way the engine already renders. You can then convert to fisheye or sample the cubemap directly in the projector correction pass.

Single projector with a fisheye lens

Choose fisheye. The display hardware already wants the circular image. A cubemap would only add a conversion step unless it is easier for your engine to render the scene that way.

Heavy shader that covers the whole dome

Usually choose a direct fisheye render. If the shader can evaluate the dome direction analytically in one pass, a cubemap can repeat the same expensive effect across multiple faces.

3D scene with strong culling and localized detail

Cubemap can be better. Each face is a conventional camera, so the engine can cull objects, manage LOD, and run familiar render passes. This is especially useful when only part of the world is detailed at any moment.

Multi-projector room with a custom calibration shader

Use either, or use both. If the calibration shader can sample cubemaps, keep the cubemap. If the mapping software expects a fulldome master, convert to fisheye. The projector warps are the final authority either way.

Video plates, 360 footage, and generated realtime layers

Hybrid is normal. Pre-rendered material may arrive as fisheye or equirectangular video, while live 3D layers render to cubemap. Composite in the direction space that loses the least detail, then output to the format the room needs.

Decision rulesA short checklist

Use these questions before choosing the intermediate: 

Do I need to deliver one file to another dome?
  choose fisheye

Am I rendering a live 3D scene with ordinary cameras?
  choose cubemap first

Can my effect compute color from a dome direction in one shader pass?
  consider direct fisheye

Does my calibration system sample cubemaps directly?
  cubemap can stay in the pipeline

Does my venue or playback stack only understand fulldome masters?
  convert to fisheye before output

The wrong choice is usually the one made implicitly. Pick the format that matches the expensive part of the job: delivery, rendering, calibration, or recording.

DebuggingWhat breaks differently

Fisheye mistakes usually show up as radial errors: rings bunch up, latitude spacing feels wrong, the center has a spoke, or the seam at 0/2π opens. Cubemap mistakes usually show up as face errors: borders do not match, a face is flipped, the top face is rotated, or screen-space effects stop at face edges. 

fisheye debug texture:
  azimuth labels around the circle
  latitude rings from center to edge
  visible seam
  center crosshair

cubemap debug texture:
  face names on every square
  arrows showing each face's up direction
  shared labels across face borders
  horizon line across the four side faces

Test with labels before testing with art. A beautiful nebula can hide a flipped face until the show is already in the room.

EpilogueThe master is not the room

A fisheye and a cubemap are both intermediates. The physical room still needs projector cameras, calibration meshes, edge blending, and color matching. That is the next visual layer.
