Dome-Native Three.js

Part three described a production pipeline in terms of a master image and projector warps. That is useful for video playback and portable fulldome content, but a realtime Three.js show does not have to start as a 2D fisheye movie.

The scene can stay live. The renderer can evaluate particles, meshes, post effects, audio reactions, and sensor input every frame, then render the result for each physical projector. You avoid baking a fisheye video. You do not avoid all texture lookups: render targets, post-processing, calibration meshes, and edge blend masks are still normal GPU tools.

This is not automatically faster. The rough comparison is: N projector renders for direct output versus one or more master renders plus N projector correction passes for the shared-master path. Direct rendering wins only when those projector views are cheaper than building a full-dome master.

Path 1 · Shared masterRender a realtime fisheye or cubemap

The conservative realtime path is still master-based, but the master is generated live. A Three.js scene renders from the dome center into a cubemap or fisheye render target. Each projector then samples that target through its calibration mesh and edge blend. 

update scene from music + sensors
render scene from dome center -> cubemap or fisheye target
for each projector:
  draw calibrated mesh
  sample the shared target
  apply edge blend and color correction

This keeps one conceptual dome image in the middle of the pipeline. It is easy to preview on a monitor, easy to record, and compatible with tools that expect fulldome masters. The cost depends on how the master is rendered: a custom fisheye pass can be one scene render, while a cubemap often means five or six scene renders before the projector passes even begin. Very wide fields of view can also make post effects harder to reason about.

Path 2 · Direct projectorsRender from each projector camera

The dome-native path treats every physical projector as a camera. Instead of first making a fisheye master, you render the Three.js scene from each projector's calibrated point of view. The output still passes through a small correction pass, but there is no single 180 degree image sitting in the middle. 

update scene from music + sensors
for each projector:
  set calibrated projector camera
  render scene to that projector's render target
  draw the projector output through warp/blend/color correction

This approach is attractive when the show is interactive, never needs to become a portable video file, and has effects that should respond to the actual projector coverage. It is not a speed optimization by default. Three projectors usually means three scene renders per frame before post-processing.

Important distinction

"No fisheye master" does not mean "no textures." A WebGL or WebGPU renderer produces images. The practical question is whether those images represent one shared dome master or one corrected output per projector.

Cost modelWhy the faster path changes

Do not count only the number of projectors. Count how much scene work happens before the final warp and blend. A useful estimate is: 

direct projector path:
  cost ~= render(P1 view) + render(P2 view) + ... + render(PN view)
          + N cheap warp/blend passes

shared master path:
  cost ~= render(one full dome master)
          + N cheap warp/blend passes

one full dome master may be:
  one custom fisheye render, or
  five or six cubemap renders

The warp and blend passes are usually not the expensive part. The expensive part is whatever your scene does: particles, shadows, raymarch steps, post effects, transparent layers, video decoding, physics-driven meshes, and sensor-reactive shaders. Direct rendering is faster only when the combined projector views are cheaper than the full-dome master.

ExamplesWhen each approach is better

One projector with a fisheye lens

Shared master is usually better. There is only one physical output, and that output often already expects a circular fisheye image. Rendering a fisheye target gives you an easy monitor preview, an easy recording, and a format that can be played back later without rebuilding the realtime scene.

Three projectors, light geometry, narrow coverage per projector

Direct projector cameras can be better. Each camera sees only part of the dome, so frustum culling and level of detail can skip work outside that projector's area. If a shared cubemap would require six large renders, three calibrated projector renders may be cheaper and easier to reason about.

Heavy full-dome shader or raymarched effect

Shared master can be better. If the effect covers the whole dome and every projector needs the same expensive result, direct rendering repeats that expensive shader once per projector. A single fisheye render followed by cheap sampling passes can avoid doing the same work three or four times.

Sensor zones tied to physical projectors

Direct projector cameras can be better. If a lidar zone, camera tracker, or performer position maps to a specific part of the room, projector-native outputs make it natural to test the response in the same coordinate frame that will be displayed. You still calibrate the output, but you do not have to keep translating every local effect through a shared fisheye image.

Touring show, dome festival, or offline render

Shared master is usually better. A fulldome master travels. The venue can use its own calibration system to map that master to its projectors. Direct projector output is more tied to the exact projector count, projector placement, and calibration data of one room.

Mixed media show with live Three.js layers

A hybrid is often the practical answer. Video plates, pre-rendered skies, and recorded segments can live in a shared fisheye or cubemap master. Local interactive particles, silhouettes, sensor effects, or projector-edge fixes can render directly into each projector output. Real systems often combine the two paths instead of treating them as exclusive.

The important thing is that "dome native" describes the coordinate system of the work, not a promise that the GPU does less work. It can do less work, the same work, or more work depending on what gets rendered repeatedly.

CalibrationThe camera is not enough

In a perfect model, a projector camera plus a mathematical dome surface would be enough. Real domes are not perfect, lenses distort, mounts shift, and projectors rarely sit exactly where the CAD drawing says. That is why direct rendering still uses calibration data. 

projector pose        -> where the projector is
projector projection  -> lens and frustum shape
warp mesh             -> measured correction on the dome
blend mask            -> overlap fade
color correction      -> match brightness and white point

The projector camera gives the first approximation. The warp mesh fixes what measurement finds. In Three.js terms, the final pass is usually a fullscreen or mesh draw that samples the projector render target and moves pixels according to calibration.

Implementation shapeA practical Three.js loop

A minimal direct-projector loop has one shared scene update and one render pass per projector. Sensor input and audio analysis happen before any projector render, so all projectors see the same world state for that frame. 

function frame(dt) {
  updateAudioClock(dt)
  updateSensors(dt)
  updateScene(dt)

  for (const projector of projectors) {
    renderer.setRenderTarget(projector.target)
    renderer.render(scene, projector.camera)

    renderer.setRenderTarget(null)
    drawWarpedOutput(projector.target, projector.mesh, projector.blend)
  }
}

If the scene is heavy and the shared master is cheap to render, the master path may be faster. If the master would require a high-resolution cubemap, or if each projector covers only a narrow part of the dome, the direct path may be comparable or faster because each camera can cull and shade less of the world. The right choice depends on scene complexity, number of projectors, master resolution, calibration workflow, and whether you need a portable fulldome master.

EpiloguePick the intermediate deliberately

The fisheye master is not mandatory. It is an intermediate representation. For realtime Three.js dome work, choose it when it helps previewing, recording, or tool compatibility. Skip it when the installation is the only target and direct projector renders make calibration or interaction simpler.
