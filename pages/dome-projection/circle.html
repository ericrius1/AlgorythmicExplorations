The Circle Under the Dome

This series is a path from a circle on paper to a full dome show: seamless visuals, calibrated projectors, music clocks, and sensor input. The first step is deliberately small. Before a dome is a room, a lens, or a show-control problem, it is a circular coordinate problem.

A rectangular screen asks for x and y. A dome asks for a direction. The easiest way to build intuition for that is the unit circle: every point can be named either by Cartesian coordinates or by a radius and an angle. Projection math is mostly the art of moving between those names without losing track of the seam.

Step 1 · Two names for one pointCartesian and polar coordinates

The left diagram shows a point in the x/y plane, with its polar name drawn on top: radius r and angle θ. The right diagram unwraps that same r and θ into a texture lookup. Drag either blue point directly: both diagrams are the same state in two coordinate systems. A fisheye dome image uses this same idea: angle around the circle becomes horizontal texture position, and distance from center becomes distance from the zenith. 

// polar -> cartesian
x = r * cos(theta)
y = r * sin(theta)

// cartesian -> polar
r     = sqrt(x*x + y*y)
theta = atan2(y, x)

atan2 is the quiet hero. A plain arctangent can tell you a slope, but it cannot tell which quadrant the point lives in. Dome mapping needs the full direction around the circle, so always reach for atan2(y, x) when recovering an angle from a vector.

Step 2 · The seam is realRadians, wraparound, and normalized UVs

Angles repeat. 0, 2π, and 4π all point to the same direction. A texture, however, has a left edge and a right edge, so the circle must be cut somewhere. That cut is the seam. Good dome content does not pretend the seam vanished; it places it where the audience will not notice, uses wrapping texture sampling, and avoids bright hard features crossing it. 

u = fract(theta / (2*pi))   // horizontal position around the circle
v = clamp(r, 0, 1)          // radial distance from center to edge

The word fract means "keep only the fractional part." It is what turns an angle that can grow forever into a repeating 0..1 texture coordinate. If the seam glitches, the first question is usually: did one side sample u = 0.001 while the other side sampled u = 0.999 as if they were far apart?

Aside · Why radians matter

Radians are not a different unit for the same old angle. They measure angle as arc length divided by radius. That makes derivatives, rotations, waves, and trig functions line up without conversion constants. Dome work is full of arcs, so radians keep the code honest.

Step 3 · Sampling is the jobFrom math to pixels

A projection mapper does not usually push content forward and hope it lands on the dome. It works backward. For each output pixel, it asks: "What direction on the dome does this pixel hit?" Then it converts that direction to a coordinate in the master image and samples there. 

for each output pixel:
  direction = pixel_to_dome_direction(pixel)
  uv        = direction_to_master_uv(direction)
  color     = sample(master, uv)

That inverse-mapping habit prevents cracks. Every output pixel receives exactly one answer. No holes appear because a source pixel failed to land somewhere, and no overlaps fight unless you intentionally create them for edge blending.

Deep dive: the center is singular

At r = 0, every angle names the same center point. That is not a bug; it is the geometry. In a shader, the center of a polar map is a singularity, so do not let code depend on a meaningful theta there. Choose any angle, or branch around the center if the effect has a visible spoke.
